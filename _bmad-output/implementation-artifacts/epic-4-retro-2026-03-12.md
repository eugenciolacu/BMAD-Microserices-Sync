## Challenges & Growth Areas

### 1. PREP-1 (ReferenceDataLoader extraction) — four-epic carryover, closed as out of scope

Epic 1's retro introduced PREP-1: extract inline reference-data loading from `ClientService/Program.cs` into `Sync.Infrastructure/Data/ReferenceDataLoader.cs`. This appeared in the Epic 2 retro as 'unclear' (❓), in the Epic 3 retro as '❌ Not done — do NOT touch Program.cs inline reference data loading,' and enters the Epic 4 retro unresolved.

Current state (confirmed): Inline reference-data loading code still exists at `ClientService/Program.cs` line 98. `AdminController.cs` carries a `// TODO: extract to ReferenceDataLoader to avoid duplication with Program.cs startup logic.` comment. Neither was touched in Epic 4 (documentation-only).

**Decision: Closed — out of scope.** This is the project's final epic. The inline logic does not affect correctness, the `README.md` references the pattern accurately, and the architectural deviation is documented inline. Extraction remains a testability improvement for a future production adaptation — not a blocking issue for the reference implementation.

Bob (Scrum Master): "Four retros, zero action. That's the data. The right decision is to name it and close it rather than carry it into a project that no longer has future epics. If someone adapts this pattern, PREP-1 is in the retrospective record as a known improvement."

Alice (Product Owner): "The TODO comment in AdminController is honest. It says 'extract to avoid duplication with Program.cs startup logic.' A developer who reads that TODO will understand exactly what to do and why."

Charlie (Senior Dev): "And to be clear: this is not a security or correctness gap. The inline logic is seeding reference-data on startup. It works. It just can't be unit-tested in its current location. For a pre-production reference implementation, that's acceptable."

### 2. PREP-4 (Integration test for `/api/v1/sync/reference-data`) — four-epic carryover, closed as out of scope

PREP-4 was first raised in the Epic 1 retro, marked `❌ Not completed` in the Epic 2 retro, flagged as escalated priority by Dana in the Epic 3 retro ('third carry — Dana escalated this'), and enters the Epic 4 retro still open. Epic 4 is documentation-only; no code or test changes were made.

Current state: 69 tests as of Epic 3 close. Zero new tests in Epic 4. The `GET /api/v1/sync/reference-data` endpoint used by ClientService on startup remains covered only by manual Docker smoke tests and the non-interrupting presence in the health scenario.

**Decision: Closed — out of scope.** This is the project's final epic. The `WebApplicationFactory`-based integration test remains a known gap. Dana's concern is documented: a latent bug in the reference-data pull path would not be caught by the automated test suite. Any production adaptation of this pattern should prioritize closing this gap.

Dana (QA Engineer): "I'll be honest — four retros and I never found time to write that test outside of a story file. The lesson is clear: if I'd raised it as a story (even a one-task story) rather than an action item, it would have had Dev Agent treatment and gotten done. That's on the process, not on any person."

Bob (Scrum Master): "Which is exactly what the Epic 3 retro's action item 3 was supposed to prevent — the promotion rule. By the time we kicked off Epic 4, the rule existed, but the mechanism to enforce it — 'create a named PREP task in sprint-status before epic start' — never happened. Epic 4 started as a documentation epic and there was no natural moment to insert a test story."

Elena (Junior Dev): "For a future project using this pattern: any retro action item that requires code change must be a real story. A note in a retro document has no sprint-status entry, no Dev Agent pickup, no completion path."

### 3. Promotion rule created but not enforced — systemic gap persists

The Epic 3 retro's Process Improvement item 3 was: 'Action items that have failed two retros become PREP stories or are closed.' The failure mode was correctly diagnosed. The rule was formally adopted. But it was not enforced before Epic 4 kickoff.

Root cause: The promotion rule requires a gatekeeping action — 'before starting Epic N, review retro items that have appeared twice and create stories or close them.' Epic 4 was a documentation epic with no natural sprint-planning session where this review would occur. The rule exists in the retro document but not in the story-generation or sprint-planning flow.

Bob (Scrum Master): "This is the hardest class of process improvement to implement: meta-process improvements. We documented the rule. We committed to the rule. We just never built the enforcement point into the workflow. The result is predictable — the rule is remembered at retrospective time, not at planning time."

Charlie (Senior Dev): "For any team using this workflow: the enforcement moment for retro meta-process rules is the SM's create-story or sprint-planning session. Not the retro. The retro is where you notice the gap; the sprint planning is where you prevent the next one."

### 4. Epic-4 administrative close pending — same pattern from prior epics

`sprint-status.yaml` shows `epic-4: in-progress` with all three stories done. This is the same pattern that was a Challenge item in the Epic 3 retro (`epic-2: in-progress` despite all stories done). The Epic 3 retro made it an action item and a team agreement: 'Epic close always includes: flip epic status to `done` and flip retrospective key to `done` in sprint-status.yaml.'

Even with that agreement in place, `epic-4` was not flipped at story completion — it is being flipped now, at retro close. The pattern persists.

Bob (Scrum Master): "The rule is right. The timing is hard. Flipping epic status feels like a minor administrative task and gets deferred. The fix requires making it part of the story-done checklist rather than the retro checklist — but the BMAD workflow's story-done flow doesn't currently enforce an epic-status check."

Alice (Product Owner): "At least the sequence is correct now: stories done → retro → flip epic status at retro close. In Epics 1 and 2 the flip was missed entirely. In Epic 3 it was caught at retro. In Epic 4 it's being caught at retro again. The error is smaller, but it's still the same error."

## Key Takeaways

1. **Documentation-only epics deliver strategic value when built on a complete implementation** — Epic 4 added no code, no tests, and no infrastructure, yet it completed the project's primary deliverable: a README that makes the sync pattern understandable, runnable, and reusable by any developer or architect.

2. **Retro action items require sprint-status entries or they will not be completed** — PREP-1 and PREP-4 survived four retrospectives unsolved. Every item that was completed (README audit, epic status flips, scope guards, story size checks) had a clear execution path inside the delivery workflow. The gap is structural, not motivational.

3. **Incremental documentation-by-story produces coherent artifacts** — Thirteen README sections across twelve stories across four epics, authored by different agents with different contexts, reads as a unified document. The key was: explicit scope guards ('this section is complete; do not re-author it'), clear insertion points ('append after line X'), and cross-references instead of duplication.

4. **The NOTE M* deviation documentation pattern is a repeatable architectural honesty convention** — When a clean-architecture principle was violated for pragmatic reasons, every instance was documented in-place with a reference chain (`// NOTE (M3.1)` → `// NOTE (M2.1)` → original). Future maintainers know the decision is intentional and consistent — not a bug or a missing import.

5. **Zero debug sessions for the final two epics confirms the compounding value of established patterns** — Epic 1 had explicit debug sessions. Epic 2 had one. Epics 3 and 4 had zero. The reduction is attributable to: scope guards designed to prevent rabbit holes, architectural conventions referenced explicitly in every story, and story sequencing designed to expose dependencies before they become blockers.
