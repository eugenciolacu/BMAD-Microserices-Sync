## Challenges & Growth Areas

1. **Story 1.4 was too large** — Domain entities (6), two DbContexts, two migration sets, a seeder, an HTTP reference-data pull client, a new controller, and DTOs all in one story. Borderline one-story too much. First DB story in a greenfield project always spikes, but the scope could have been split: entities + migrations in one story, seeder + HTTP pull in a follow-up.

2. **SQLite vs. SQL Server concurrency model divergence** — `byte[] RowVersion` on SQL Server vs. `long ConcurrencyStamp` on SQLite requires mental context-switching. The pattern works for reference entities but the translation strategy across provider boundaries is not yet documented for Measurement sync in Epic 2.

3. **Partial extraction of `ReferenceDataLoader`** — The reference-data pull logic lives inline in `ClientService/Program.cs` (startup) and was partially extracted for the `/api/v1/admin/pull-reference-data` endpoint in Story 1.5. Two copies of similar logic; not fully DRY.

4. **DTO mirroring** — `ReferenceDataDto` defined in `ServerService/Models/Sync/` and `SyncReferenceDataDto` defined in `ClientService/Models/Sync/`. Same shape, two files, no shared home. Compile-time risk if fields diverge.

5. **HTTP reference-data pull path has no automated test coverage** — The `GET /api/v1/sync/reference-data` → ClientService startup pull path is manually verified only. Integration test gap.

---

## Key Insights & Lessons Learned

1. **"What NOT to do" is a first-class quality tool.** It sounds like documentation nicety but it's a scope guard that actively prevented five potential scope creeps. Keep it. Enforce it in every story's Dev Notes going forward.

2. **First DB story in a greenfield project always spikes — plan for it.** Story 1.4 was a known cost, not a process failure. The right response is to size it honestly upfront and resist adding more scope mid-story. Subsequent stories won't have the same spike.

3. **Partial extractions are debt from the moment they're partial.** Complete the extraction in the same story or document it as a named debt item with an owner. No orphaned half-refactors left untracked.

4. **The concurrency token divergence (SQL Server `rowversion` vs. SQLite `ConcurrencyStamp`) is the deepest technical seam in this project.** It needs a documented translation strategy before Epic 2's sync stories (2.2, 2.3) touch it under implementation pressure.

5. **Manual Docker smoke tests are a meaningful quality gate.** Eugen's manual verification of Story 1.4 on 2026-03-05 caught real end-to-end confidence. Formalise this as a post-story ritual.
