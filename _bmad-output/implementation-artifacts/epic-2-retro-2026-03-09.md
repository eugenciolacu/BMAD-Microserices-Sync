## Successes & Strengths

### 1. Clean execution velocity with a single debug cycle
Stories 2.1 through 2.4 had zero debug sessions in their Dev Agent Records. Story 2.5 had exactly one (FK seeding in the repeatable-run test), caught and fixed immediately. In an epic that touched HTTP clients, SQLite transactions, SQL Server transactions, EF bulk updates, and cross-service DTOs, that is a remarkably clean run.

Bob (Scrum Master): "One debug session across five stories that each touched multiple layers. That's not luck — that's the payoff of investing in good story notes and established patterns."

### 2. `// NOTE (M2.x)` deviation pattern maintained with honesty
The acknowledged direct-DbContext injection pattern from Epic 1's `// NOTE (M4)` was extended consistently: `// NOTE (M2.1)` for `MeasurementGenerationService`, `// NOTE (M2.2)` for `MeasurementSyncService`. The team didn't silently accrue debt — they documented it in-place and referenced the same justification chain.

Charlie (Senior Dev): "This is important. We know those deviations exist, we know why they exist, and we know where they are. That's a much better position than cleaner-looking code with hidden surprises."

### 3. `ExecuteUpdateAsync` resolved the RowVersion/ConcurrencyStamp tension elegantly
Epic 1's PREP-3 flagged the `byte[] RowVersion` (SQL Server) vs. `long ConcurrencyStamp` (SQLite) divergence as a risk needing a documented translation strategy before Story 2.2. Instead of writing the strategy doc, Story 2.2 made the problem architectural: `ExecuteUpdateAsync` for bulk `SyncedAt` updates bypasses EF change tracking entirely, so the concurrency token never enters the equation for that operation. The problem was dissolved rather than solved.

Alice (Product Owner): "I love when that happens."

### 4. Scope guard discipline held across all five stories
Every story's Dev Notes contained explicit "STOP HERE" / "What NOT to do" scope prohibitions — and every completion note confirmed they were respected. Most notably, the jqGrid CRUD endpoints (`GetPaged`, `GetById`, `Create`, `Update`, `Delete`) were explicitly prohibited in Stories 2.1, 2.2, and 2.3, and none were added. Epic 3 story 3.2 inherits a clean slate.

Dana (QA Engineer): "Those prohibitions aren't just documentation niceties. In Story 2.3, the note 'do NOT delete existing measurements before a pull' prevented a real correctness bug — delete-and-reinsert would have wiped the client's `SyncedAt` values and broken the whole semantics."

### 5. Test infrastructure established for the rest of the project
By end of Story 2.4, the test project had: `TestableServerDbContext` (SQLite-compatible server override), `StubHttpClientFactory` + `MockHttpMessageHandler` (no Moq), full FK seeding chains, `DatabaseResetter` integration, and `IDisposable` + open `SqliteConnection` teardown — all established, all documented in cross-story references. Story 3.x test files will be able to reuse these patterns directly.

### 6. Single round-trip HTTP pattern enforced (not multiple requests per batch)
A subtle architectural win: the client sends ALL pending measurements in a single POST, and the server internally chunks into in-memory batches inside one transaction. This was explicitly enforced in story notes. The alternative — one POST per batch — would have caused partial-visibility windows and made rollback semantics much harder.

## Challenges & Growth Areas

### 1. PREP-1 (Extract `ReferenceDataLoader`) — status unclear
Epic 1's retro committed to extracting the `ReferenceDataLoader` inline logic from `ClientService/Program.cs` into `Sync.Infrastructure/Data/ReferenceDataLoader.cs` before Story 2.1. None of the Epic 2 story completion notes reference this extraction, and the Story 2.2 reference file still points to `AdminController.cs` as the direct-DbContext pattern model (implying the inline logic may still be there). Whether this was done outside of story context or quietly deferred is unknown.

Bob (Scrum Master): "This is a question mark we need to resolve before Epic 3. Not because it blocks Story 3.x, but because tech debt we can't locate is the most dangerous kind."

Elena (Junior Dev): "Should we just check the file?"

Bob (Scrum Master): "Yes — that's an action item."

### 2. PREP-2 (Consolidate Sync DTOs) — resolved via naming convention, not shared location
Epic 1's retro committed to consolidating DTOs into `Sync.Application/Models/Sync/`. What actually happened: Story 2.2 added `ClientService/Models/Sync/MeasurementPushDto.cs` (internal sealed `ClientMeasurementPush*` types) and Story 2.3 added `ClientService/Models/Sync/MeasurementPullDto.cs`. The collision problem was solved by the `Client` prefix naming convention, keeping ServerService DTOs in `ServerService/Models/Sync/` and ClientService DTOs in `ClientService/Models/Sync/`. This is a workable pattern but it means the Epic 1 commitment was not executed as specified.

Charlie (Senior Dev): "Honestly, I think what we ended up with is better for this project. `internal sealed` DTOs scoped to each service, no shared Application layer dependency from infrastructure. The naming convention is clear enough."

Alice (Product Owner): "I'll defer to Charlie on this one — but we should at least acknowledge we made that call rather than pretend the original plan was followed."

### 3. PREP-4 (Integration test for `/api/v1/sync/reference-data`) — not done
Epic 1 committed to a WebApplicationFactory-based integration test for the reference-data pull endpoint before Story 2.2. No completion note in any Epic 2 story confirms this was done. The HTTP reference-data pull path from ClientService startup through `GET /api/v1/sync/reference-data` on ServerService remains manually verified only.

Dana (QA Engineer): "This one matters. It's not a blocking gap — the unit tests around the downstream behavior are solid — but a full integration test for that bootstrap path would catch a whole class of configuration or startup bugs."

### 4. `MeasurementsController` growing beyond a focused controller
By end of Story 2.4, `ClientService/Controllers/MeasurementsController.cs` injects: `MeasurementGenerationService`, `MeasurementSyncService`, `ClientDbContext`, and `IHttpClientFactory`. Story 3.2 will add jqGrid CRUD endpoints which will likely add more injection. The controller started as a single-responsibility class in Story 2.1 and has accumulated scope over four stories.

Charlie (Senior Dev): "It's not a God object yet. But Story 3.2 is the decision point — if we add a `MeasurementQueryService` or similar for jqGrid, we could keep the controller thin. If we just dump the jqGrid logic in the controller, we lose that window."

Bob (Scrum Master): "Good catch. Flagging for an Epic 3 pre-story conversation."

### 5. No correlation ID scaffolding for future structured logging
Story 3.4 requires structured logs with correlation IDs per sync run. Epic 2's logging is `ILogger<T>` with natural-language messages — no correlation IDs, no structured context. When Story 3.4 arrives, it will need to retrofit correlation context into all sync operations, not just add new log lines. The cost of not setting this up early is higher than it would have been if added during Story 2.2.

Dana (QA Engineer): "Each story had logging, but it was all plain-text. Coming back to thread correlation IDs through `MeasurementSyncService.PushAsync`, `PullAsync`, `SyncMeasurementsController.Push`, `Pull` — that's more invasive work in Epic 3."

## Epic 3 Preview — Sync Diagnostics and Audit

**Goal:** Enable developers to see what happened during sync: view summaries of sync runs, inspect data via jqGrid, connect directly to databases, and follow structured logs to diagnose unexpected behavior.

**Stories:** 3.1–3.5 | **FRs:** FR10, FR11, FR12

Bob (Scrum Master): "Let me walk through what's coming and what Epic 2 leaves us ready — or not ready — for."

### Dependencies on Epic 2 (all confirmed ready):
- ✅ Measurements push/pull pipeline complete and stable
- ✅ `SyncedAt` semantics documented (client-side: "pushed by this client"; server: always null)
- ✅ 5-client sync topology verified working
- ✅ Reset + generate + push + pull cycle verified repeatable (Story 2.5)
- ✅ `GET /api/v1/sync/measurements/count` endpoint available (Story 2.4)

### Gaps and preparation needed for Epic 3:

**Story 3.1 (Sync Run Summary View)** — will need a new `SyncRun` entity on ServerService with its own SQL Server migration. No `SyncRun` table was introduced in Epic 2. The existing `Measurement` table records `UserId`, `RecordedAt` — so per-client per-run summaries can be derived — but a proper summary view may want a dedicated `SyncRuns` table with `ClientId`, `Timestamp`, `RunType` (push/pull), `Count`, `Status`.

**Story 3.2 (jqGrid Measurement Inspection)** — `MeasurementsController` on ClientService needs jqGrid paging/sorting endpoints. These were explicitly deferred in Stories 2.1–2.3. The decision about whether to extract a `MeasurementQueryService` or add it directly to the controller should be made before the story is written, not discovered during dev.

**Story 3.4 (Structured Logging)** — correlation IDs will need to be threaded through `PushAsync`, `PullAsync`, and both controllers. This is more invasive than adding new log lines. It should be scoped carefully in the Story 3.4 spec.

Charlie (Senior Dev): "Story 3.1 is the one I'd want to plan carefully. If we add a `SyncRuns` table, that's a new domain entity, a new migration, a new controller, and potentially a new mapping layer. It could be another Story 1.4-sized story if we're not careful."

Alice (Product Owner): "And it's Story 3.1 — the very first story of the epic. We've learned to flag those."

Bob (Scrum Master): "Good. That's now on the pre-story checklist."

## Key Takeaways

1. **The patterns established in Epic 1 compounded in Epic 2.** Thin `Program.cs`, `// NOTE (M)` acknowledged deviations, scope guards, smoke tests as rituals — all of these led to a near-zero-debug epic. The investment in architecture and patterns in Epic 1 paid off here, and the same will be true going into Epic 3.

2. **PREP tasks without a named tracking entry don't get done.** PREP-2 and PREP-4 from the Epic 1 retro were not completed as specified. The lesson is not to do more prep — it is to make prep work visible and owned in the same system as stories. Anonymous prose in a retro document is not a work-tracking system.

3. **`ExecuteUpdateAsync` as a concurrency bypass is the right pattern for this project's sync semantics.** The RowVersion / ConcurrencyStamp translation concern from PREP-3 never materialized as a problem in Epic 2 because the implementation pattern dissolved it. This is worth documenting formally in `architecture.md` so future stories don't reopen the question.

4. **Story 3.1 is the Epic-1.4 candidate for Epic 3.** A new domain entity (`SyncRun`), new SQL Server migration, new controller, and new UI grid in a single story is the same multi-layer profile that made Story 1.4 large. Flag it before creation.

5. **The test infrastructure is now a genuine project asset.** `TestableServerDbContext`, `StubHttpClientFactory`, `MockHttpMessageHandler`, full FK seeding helpers — documented across 5 stories, proven across 28 tests. Epic 3 test authors should read `MeasurementPullTests.cs` and `RepeatableSyncRunTests.cs` as their starting points.
